= CCFS File System Specification
:sectnums:

== Introduction

This document describes the on-storage format of the Compass Computer File
System used by GRiD-OS / CCOS on the GRiD Compass. CCFS is used by the Compass
both on internal Bubble Memory and on external disks.

The specification is based on observed GRiD-OS images, available CCOS
filesystem tooling and field names found in InteGRiD source references. It is
intended to be sufficient for an independent compatible reader/writer. Fields
whose full meaning is not known are preserved in the layout and marked as
unknown.

[[notation]]
== Notation

Offsets are zero based and relative to the start of the described structure
unless stated otherwise. Numbers with `0x` are hexadecimal. Other numbers are
decimal. Multi-byte integer fields are little-endian. Structures are byte
packed.

This document uses the term sector for the filesystem allocation unit. Some
field and structure names inherited from source references use legacy names;
those names are kept as identifiers.

[[hierarchy-and-names]]
== Hierarchy and Names

CCFS is hierarchical, but GRiD-OS does not support arbitrary nested folders in
the user-visible model. The expected hierarchy is:

[source,text]
----
Device -> Subject -> File
----

A subject is the CCOS directory in GRiD-OS / CCOS terms. Subject files use the
type `Subject`.

The root directory name is used as the partition name. On a Compass,
the user interface displays this name when it is present. If the root
directory name is empty, the interface falls back to a default storage name,
for example `Hard Disk` or `Portable Floppy`, depending on the device.

Names are length-delimited byte strings, not NUL-terminated strings. Normal
names use this form:

[source,text]
----
BaseName~Type~
----

Examples:

[source,text]
----
Programs~Subject~
Executive~Run~
TypeGRiD5x8~Font~
----

The first `~` separates the base name from the type. The final `~` terminates
the type and is part of the stored name.

`Run` files can include a target file type after a space:

[source,text]
----
BaseName~Run Font~
----

In this example, opening a `Font` file causes GRiD-OS to launch this `Run`
binary as the handler for that file type.

Passwords, when encoded in the visible name, are appended after a `|` character
at the end of the file name.

[[sector-sizes]]
== Sector Sizes

CCFS supports two sector-size variants for two storage classes. Internal
Compass Bubble Memory uses 256-byte sectors. External disk and floppy media use
512-byte sectors. Structures whose usable area, array length, padding or
checksum range differs between these media variants are marked in the relevant
section.

[[boot-sector]]
== Boot Sector

The first 512 bytes of a CCFS image contain a boot sector.

The boot sector is partially compatible with the IBM PC MBR layout: it reserves
the conventional partition table area at offset 446 and the final signature
word at offset 510. CCFS also uses Compass-specific fields before the boot code
area to locate the allocation bitmap and the superdirectory.

On GRiD-OS media, 2048 bytes of the additional boot code follow immediately
after the 512-byte boot sector region.

[[superdirectory]]
== Superdirectory

The superdirectory is the root directory of the filesystem. Its inode sector is
the filesystem entry point for directory traversal. It is a normal CCFS
directory in its on-storage structure; its only structural special case is that
its parent directory id points back to itself:

[source,text]
----
desc.dir_file_id == header.file_id == superblock_fid
----

Its payload follows the normal directory stream format; see
<<directory-structure>>.

The preferred location source is `ccos_boot_sector_t.superblock_fid`. If that
field is zero or the boot sector does not provide usable filesystem entry
points, GRiD-OS relies on media parameters returned by the disk Status response.

For external disk and floppy devices, the fallback entry sectors are provided
by device firmware through the Status response:

* `0x0121` for smaller 512-byte external disk or floppy images.
* `0x2420` for larger 512-byte hard disk images.

For 256-byte Bubble Memory images, the observed superdirectory sector is
`0x03fe`.

[[allocation-bitmap]]
== Allocation Bitmap

The allocation bitmap tracks which sectors are occupied in the volume. A bit
value of 1 means the sector is occupied. A bit value of 0 means the sector is
free. A writer must update it whenever it allocates or frees an inode, content
inode, payload sector or bitmap sector.

The preferred location source is `ccos_boot_sector_t.bitmap_fid`, which points
to the first bitmap sector. If that field is zero or the boot sector does not
provide usable filesystem entry points, GRiD-OS relies on media parameters
returned by the disk Status response, as it does for the superdirectory.

For external disk and floppy devices, the fallback first bitmap sector is
provided by device firmware through the Status response:

* `0x0120` for 512-byte floppy images.
* `0x2400` for 512-byte hard disk images.

For 256-byte Bubble Memory images, the observed first bitmap sector is
`0x03fd`.

On Bubble Memory, one bitmap sector stores 248 bitmap bytes and covers 1984
filesystem sectors. On external disk/floppy media, one bitmap sector stores 500
bitmap bytes and covers 4000 filesystem sectors.

The amount of storage occupied by the bitmap is:

[source,text]
----
volume_sector_count       = volume_size / sector_size
bitmap_bytes_per_sector   = 248 for 256-byte media, 500 for 512-byte media
sectors_per_bitmap_sector = bitmap_bytes_per_sector * 8
required_bitmap_bytes     = ceil(volume_sector_count / 8)
bitmap_sector_count       = ceil(required_bitmap_bytes / bitmap_bytes_per_sector)
bitmap_storage_bytes      = bitmap_sector_count * sector_size
----

To test whether a sector is free or occupied:

[source,text]
----
bitmap_sector_index = sector_number / sectors_per_bitmap_sector
local_sector        = sector_number % sectors_per_bitmap_sector
byte_index          = local_sector / 8
bit_index           = local_sector % 8
mask                = 1 << bit_index
bitmap_sector_number = bitmap_fid + bitmap_sector_index
bitmap_sector        = read_sector(bitmap_sector_number)
bitmap_byte          = bitmap_sector.bytes[byte_index]
is_occupied          = (bitmap_byte & mask) != 0
----

`bit_index` 0 is the least significant bit of the byte.

When allocating a sector, set the corresponding bit to 1 and increment the
`allocated` field in every bitmap sector. When freeing a sector, clear the bit
to 0 and decrement the `allocated` field in every bitmap sector.

[[inode-and-sector-structures]]
== Inode and Sector Structures

Every file and directory is represented by an inode sector. The inode sector
number is also the file id used by directory entries, payload sectors and
content inode sectors.

Most CCFS sectors begin with a 4-byte owner header. Inodes then add a metadata
descriptor and the first payload sector-reference list. Large files extend that
list through content inode sectors.

=== Content Inode Sectors

A file's payload sector list starts in the main inode. If the embedded inode
array is full, additional sector numbers are stored in content inode sectors.
Content inode sectors form a linked list through the `block_next`,
`block_current` and `block_prev` fields of `ccos_block_data_t`.

Sector-reference fields use `0xffff` as the invalid sector reference value. It
acts as a null pointer in chain links and marks unused entries in
sector-reference arrays.

==== `ccos_content_inode_t`

Nominal structure size: 512 bytes.

A content inode is an extension node for a file's payload sector-number list.
It does not contain file data.

[%header,cols="2,1,1,5"]
|===
| Name
| Offset
| Size
| Comment

| `content_inode_info`
| 0
| 12
| Sector-list header for this extension node.

| `sector_refs`
| 12
| Variable
| Additional payload sector-number array. Unused entries are normally
`0xffff`.

| `padding`
| End
| Variable
| Unused bytes excluded from the content inode checksum.
|===

Media-specific layout:

* 256-byte sectors: references at `12..251`, 120 references, padding at
  `252..255`.
* 512-byte sectors: references at `12..503`, 246 references, padding at
  `504..511`.

The chain starts at `inode.content_inode_info.block_next`. Each node points to
the next node using its `content_inode_info.block_next`. `0xffff` terminates
the chain. The `block_current` and `block_prev` fields should match the
physical chain links and are useful for validation and reverse traversal.

[[directory-structure]]
== Directory Structure

Directories are ordinary files whose payload is a marker-delimited directory
stream. The inode descriptor field `dir_count` stores the expected number of
entries. Directory entries are sorted case-insensitively by parsed base name
first, then by parsed type.

=== Directory Entry

Directory entries are variable-length records inside directory file data.

Fixed entry prefix size: 3 bytes.

[%header,cols="2,1,1,5"]
|===
| Name
| Offset
| Size
| Comment

| `sector`
| 0
| 2
| Inode sector number/file id of the referenced file.

| `name_length`
| 2
| 1
| Number of file name bytes following the fixed prefix.
|===

Full serialized entry, excluding the marker byte before it:

[%header,cols="2,1,1,5"]
|===
| Name
| Offset
| Size
| Comment

| `sector`
| 0
| 2
| Inode sector number/file id of the referenced file.

| `name_length`
| 2
| 1
| Number of file name bytes in `name`.

| `name`
| 3
| `N`
| Raw file name bytes copied from the referenced inode descriptor.

| `reverse_length`
| `3 + N`
| 1
| Entry length excluding the following marker byte. Must be `N + 4`.
|===

=== Directory Stream

Directory payload is a marker-delimited stream:

[source,text]
----
marker0, entry0, marker1, entry1, marker2, ..., entryN, markerN+1
----

Marker values:

* `0x00`: an entry follows this marker.
* `0xff`: end of directory. No entry follows this marker.

An empty directory payload is one byte:

[source,text]
----
ff
----

For each entry:

[source,text]
----
entry_size_without_following_marker = name_length + 4
serialized_size_including_marker    = name_length + 5
reverse_length                      = name_length + 4
----

=== Parsing a Directory

Let `data` be the directory payload and `size` be its byte length.

Directory parsing starts at offset 0:

. Read one marker byte.
. If the marker is `0xff`, stop.
. If the marker is not `0x00`, the directory stream is malformed.
. Read a directory entry at the following offset.
. Let `N = name_length`.
. Verify that `reverse_length == N + 4`.
. Resolve `sector` as an inode sector number.
. Advance by `N + 4` bytes to the next marker.
. Repeat until `0xff` or until `dir_count` entries have been read.

A robust implementation should check all bounds before reading entry fields.

[[files-in-directories]]
== Files in Directories

A directory stores references to child files in its directory stream. Each
directory entry contains the child inode sector number and a copy of the child
name. The child inode also stores its parent directory in
`desc.dir_file_id`.

For a consistent image:

* the directory entry `sector` should point to the child inode sector;
* the child inode `header.file_id` should equal that same sector number;
* the child inode `desc.dir_file_id` should equal the parent directory inode
  sector;
* the directory entry name should match the child inode descriptor name;
* `dir_count` should match the number of entries in the directory stream.

The directory stream does not contain file data. It only names children and
points to their inode sectors. File contents are reached from the child inode.

[[file-contents]]
== File Contents

File contents are stored in payload sectors. The file inode stores the first
payload sector references directly. If a file needs more payload sectors than
fit in the inode, the remaining references are stored in content inode sectors.

The logical file size is normally `inode.desc.file_size`. For directory files,
readers should use `inode.desc.dir_length` when it differs from
`inode.desc.file_size`, because some older or observed images have root or
directory `file_size` values larger than the actual directory byte stream.

=== File Payload Sector

Payload sectors have a fixed layout.

[%header,cols="2,1,1,5"]
|===
| Name
| Offset
| Size
| Comment

| `header`
| 0
| 4
| `file_id` is the owner inode id. `file_fragment_index` is the zero-based
payload sector index within the file.

| `data`
| 4
| Variable
| File data bytes.

| `unused_tail`
| End
| Variable
| On 512-byte media, the logical payload area ends at byte 507 and bytes
`508..511` are not file data.
|===

Media-specific payload area:

* 256-byte sectors: header `0..3`, data `4..255`, no ignored tail. The logical
  payload size is 252 bytes.
* 512-byte sectors: header `0..3`, data `4..507`, ignored tail `508..511`. The
  logical payload size is 504 bytes.

=== Sector Reference Order

The payload sector list is read in this order:

. inode embedded `sector_refs`, in array order;
. first content inode referenced by `inode.content_inode_info.block_next`;
. each later content inode reached through `content_inode_info.block_next`;
. each content inode `sector_refs` array, in array order.

Entries equal to `0xffff` are unused. For compatibility with observed images,
readers should ignore `0xffff` entries wherever they appear in the arrays.
Readers should also tolerate payload sector references outside the image during
read-only enumeration.

The payload sector count needed for a file is:

[source,text]
----
payload_sector_count = ceil(file_size / logical_payload_size)
----

where `logical_payload_size` is 252 for 256-byte media and 504 for 512-byte
media.

=== Large Files

A file starts with the embedded sector-reference array in the inode. Large
files continue through a linked list of content inode sectors.

[source,text]
----
direct_refs             = 20 for 256-byte media, 148 for 512-byte media
refs_per_content_inode  = 120 for 256-byte media, 246 for 512-byte media
payload_size            = 252 for 256-byte media, 504 for 512-byte media
payload_sector_count    = ceil(file_size / payload_size)
extra_ref_count         = max(0, payload_sector_count - direct_refs)
content_inode_count     = ceil(extra_ref_count / refs_per_content_inode)
----

There is no separate tree or extent structure for very large files. They keep
using the same linear content inode chain until the file is complete or the
volume runs out of sectors. The file size field is 32-bit, while individual
sector references are 16-bit; the practical maximum file size is therefore
limited by both the volume sector count and the number of available metadata
sectors for the content inode chain.

=== Reading File Data

To read a file:

. Load the inode at the file id sector.
. Build the payload sector list using the order above.
. Read each payload sector from offset 4.
. Concatenate payload bytes in sector-reference order.
. Validate that each payload sector `file_fragment_index` matches its zero-based
  position when strict validation is required.
. Return at most the logical file size.

[[modification-algorithms]]
== Modification Algorithms

A writer must keep these structures consistent after every change:

* inode descriptor fields such as `file_size`, `dir_length` and `dir_count`;
* directory entries in the parent directory stream;
* inode and content inode sector-reference lists;
* payload sector headers;
* allocation bitmap bits and `allocated` counts;
* metadata, sector-list, content inode and bitmap checksums.

=== Allocate or Free a Sector

To allocate a sector, find a zero bit in the allocation bitmap, set it to 1,
increment `allocated` in every bitmap sector and recalculate affected bitmap
checksums. To free a sector, clear its bitmap bit, decrement `allocated` in
every bitmap sector and recalculate affected bitmap checksums.

=== Create a Directory

To create a directory:

. Validate the directory name as a length-delimited CCFS name.
. Allocate one inode sector for the directory.
. Allocate one payload sector for the directory stream.
. Initialize the payload sector with a single `0xff` end marker.
. Initialize the inode as a directory: set `file_size` and `dir_length` to the
  directory stream length, set `dir_count` to 0 and set `dir_file_id` to the
  parent directory inode sector.
. Store the payload sector number in the directory inode's first
  `sector_refs` entry and fill unused entries with `0xffff`.
. Add a directory entry to the parent directory stream.
. Recalculate parent and child checksums.

=== Delete a Directory

To delete a directory:

. Load and parse the directory stream.
. Refuse deletion if the directory contains entries.
. Remove the directory entry from the parent directory stream.
. Free all payload sectors referenced by the directory inode.
. Free any content inode sectors if the directory stream had grown beyond the
  inode embedded references.
. Free the directory inode sector.
. Update parent `dir_length` and `dir_count`.
. Recalculate affected checksums.

The superdirectory must not be deleted.

=== Create a File

To create a file:

. Validate the file name as a length-delimited CCFS name.
. Allocate one inode sector for the file.
. Allocate the payload sectors required for the file contents.
. Initialize each payload sector header with the file id and zero-based payload
  sector index.
. Store payload sector numbers in the inode embedded references and then in
  content inode sectors as needed.
. Set `file_size`, `name_length`, `name`, timestamps, parent directory id and
  other descriptor fields.
. Add a directory entry to the parent directory stream.
. Recalculate metadata, sector-list, content inode, parent directory and bitmap
  checksums.

=== Delete a File

To delete a file:

. Remove the file entry from the parent directory stream.
. Free all payload sectors referenced by the inode and content inode chain.
. Free all content inode sectors.
. Free the inode sector.
. Update parent `dir_length` and `dir_count`.
. Recalculate affected checksums.

[[checksum-algorithm]]
== Checksum Algorithm

All checksums are 16-bit additive checksums over little-endian words.
The covered byte ranges are documented in the comments for the corresponding
checksum fields: `checksum`, `blocks_checksum` and `metadata_checksum`.

Algorithm:

[source,text]
----
sum = 0
for each pair of bytes lo, hi:
    sum = (sum + (hi << 8) + lo) & 0xffff
if one byte remains:
    sum = (sum + byte) & 0xffff
return sum
----

[[volume-formatting]]
== Volume Formatting

Existing CCOS images may differ, but these rules describe the normal formatted
layout expected by GRiD-OS-compatible media.

Disk image sizes are normally multiples of 512 bytes. This also applies to
256-byte Bubble Memory images.

Every sector is initialized as:

[source,text]
----
bytes 0..3   = ff ff ff ff
bytes 4..end = 55
----

For a general implementation, compute the minimum bitmap size as:

[source,text]
----
volume_sector_count = volume_size / sector_size
required_bytes      = (volume_sector_count + 7) / 8
bytes_per_bitmap    = 248 for 256-byte media, 500 for 512-byte media
bitmap_count        = (required_bytes + bytes_per_bitmap - 1) / bytes_per_bitmap
----

A formatter should create consecutive bitmap sectors starting at `bitmap_fid`.
In the last bitmap sector, unused bits after the last volume sector should be
set to 1.

Some existing formatted images use this simpler calculation:

[source,text]
----
sector_count     = disk_size / sector_size
required_bytes   = sector_count / 8
bitmap_count     = required_bytes / bytes_per_bitmap + 1
tail_offset      = required_bytes % bytes_per_bitmap
tail_length      = bytes_per_bitmap - tail_offset
----

After bitmap initialization, mark the bitmap sectors themselves as occupied.

The superdirectory inode is written at `superblock_fid`. Formatters normally
zero the whole superdirectory inode sector first; fields not mentioned below
remain zero unless changed by checksum calculation.

Initial superdirectory fields:

* `header.file_id = superblock_fid`
* `header.file_fragment_index = 0`
* `desc.file_size = 252` on 256-byte media, `504` on 512-byte media
* `desc.dir_length = 0` in some observed formatted images; logically, an empty
  directory stream contains one byte (`0xff`), so a stricter formatter should
  set this to 1
* `desc.dir_count = 0`
* `desc.name_length = 0`
* `desc.name` is space-filled
* `desc.creation_date` is the current date/time returned by the embedding
  application
* `desc.mod_date = desc.creation_date`
* `desc.expiration_date` is all zero bytes
* `desc.dir_file_id = header.file_id`
* `desc.protec = 1`
* `desc.pswd_len = 5`
* `desc.pswd = 29 ff 47 c7`
* `content_inode_info.header.file_id = superblock_fid`
* `content_inode_info.header.file_fragment_index = 0`
* `content_inode_info.block_next = 0xffff`
* `content_inode_info.block_current = superblock_fid`
* `content_inode_info.block_prev = 0xffff`

The superdirectory inode embedded sector-reference list is filled with
`0xffff`, then entry 0 is set to `superblock_fid + 1`. That sector is
initialized as the superdirectory payload sector. Its payload starts with the
directory last-entry marker `0xff`, making the superdirectory initially empty.

[[validation-recommendations]]
== Validation Recommendations

A robust implementation should validate at least:

* boot sector `bitmap_fid` and `superblock_fid` are within the image;
* superdirectory inode `header.file_id == superblock_fid`;
* inode `header.file_id == content_inode_info.header.file_id`;
* sector references are either `0xffff` or within the image sector count;
* content inode chains do not cycle;
* directory entry reads stay within `dir_length`;
* directory `reverse_length` equals `name_length + 4`;
* unused bitmap bits after the last volume sector are set;
* bitmap `allocated` agrees with counted occupied bits;
* checksums match where present.

[[known-unknowns]]
== Known Unknowns

The following descriptor fields are preserved and checksummed, but their full
meaning is not yet documented here:

`machine_ID`, `comp`, `encry`, `protec`, `pswd_len`, `pswd`, `asc`, `system`,
`rom`, `rom_id`, `mode`, `RDB`, `UDB`, `grid_central_use`.

Future work should confirm their meanings from original GRiD/InteGRiD
documentation or from a larger corpus of known-good images.
